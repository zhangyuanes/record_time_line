# 03 实现计划

## 上下文
基于 `01_requirement.md` 与 `02_interface.md`，本计划聚焦逻辑拆分与执行顺序，不限定唯一技术实现。  
前端体验优先，后端服务以稳定本地读写与导出可靠性为核心。

## 分步执行

### 步骤 1：初始化工程与目录规范
- 文件：`docs/specs/chronos-canvas/`（已完成）
- 操作：确认规格文件冻结为当前版本。
- 逻辑：
  - 约定项目目录结构（示例）：
    - `apps/web`（前端编辑与预览）
    - `apps/local-server`（本地 API 与导出服务）
    - `projects`（用户本地项目数据根目录）
  - 明确环境变量与端口规划，避免前后端冲突。
- 验证：
  - 可启动前端与本地服务。
  - 空目录下可成功初始化第一个项目。

### 步骤 2：定义数据模型与持久化协议
- 文件：`apps/local-server/src/domain/*`（建议）
- 操作：创建 `ProjectMeta`、`TimelineDocument`、`TimelineNode` 等模型与校验器。
- 逻辑：
  - 建立 `data.json` 版本号与迁移入口（`version`）。
  - 所有媒体引用落地为相对路径。
  - 增量保存时更新 `updatedAt`，保证可追踪。
- 验证：
  - 新建项目后自动生成合法 `data.json`。
  - 手工篡改数据可触发校验失败并返回可读错误信息。

### 步骤 3：实现项目管理 API
- 文件：`apps/local-server/src/modules/projects/*`
- 操作：实现项目创建、查询、更新、删除（或归档）接口。
- 逻辑：
  - 扫描 `projects/` 目录生成项目列表。
  - `slug` 冲突处理（自动追加后缀）。
  - 目录与元数据保持一致性。
- 验证：
  - 创建 3 个项目后可正确列表与切换。
  - 非法 `projectId` 返回 `PROJECT_NOT_FOUND`。

### 步骤 4：实现节点管理与排序 API
- 文件：`apps/local-server/src/modules/nodes/*`
- 操作：实现节点增删改查、批量重排。
- 逻辑：
  - 节点按 `timestamp` 与手动排序规则共同决定展示顺序。
  - 支持 `layout`、`highlight`、`type` 等展示字段。
  - 保存操作支持幂等（重复提交不产生脏数据）。
- 验证：
  - 新增/编辑/删除节点后刷新页面仍一致。
  - 重排后导出内容顺序与预览一致。

### 步骤 5：实现媒体上传与资源管理
- 文件：`apps/local-server/src/modules/assets/*`
- 操作：实现上传、删除、引用计数（可选）与清理策略。
- 逻辑：
  - 上传后返回 `MediaAsset`，写入项目 `assets/`。
  - 图片读取宽高用于占位，降低布局跳变（CLS）。
  - 可选增强：生成缩略图与压缩版本。
- 验证：
  - 上传图片/视频后节点可立即引用并预览。
  - 删除已引用资源时有明确拦截或替换策略。

### 步骤 6：构建前端编辑工作台（高优先级）
- 文件：`apps/web/src/features/editor/*`
- 操作：完成编辑区、节点表单、时间线画布、交互反馈。
- 逻辑：
  - 支持快速新增节点、表单校验、即时保存。
  - 支持拖拽定位与单节点聚焦查看。
  - 引入统一状态管理，避免预览与编辑状态漂移。
- 验证：
  - 编辑动作后 1 秒内预览同步。
  - 长列表编辑中无明显输入卡顿或状态错乱。

### 步骤 7：实现主题系统与动效策略
- 文件：`apps/web/src/features/theme/*`
- 操作：实现主题配置读取、主题切换、动效参数化。
- 逻辑：
  - 至少实现 3 套主题：`minimal-list`、`snake-path`、`horizontal-film`。
  - 动效参数（时长、缓动、触发）从配置读取，不在组件硬编码。
  - 为低性能设备提供 `motionLevel` 降级策略。
- 验证：
  - 主题切换后布局与动效变化明显，数据不变。
  - `motionLevel=low` 时仍保持可用且不卡顿。

### 步骤 8：实现导出服务（核心交付）
- 文件：`apps/local-server/src/modules/export/*`
- 操作：生成离线分享包并拷贝资源。
- 逻辑：
  - 生成导出目录：`index.html` + `assets/` + `data.json`。
  - 重写运行时路径，确保 `file://` 打开可用。
  - 禁止产物包含本机绝对路径。
  - 可选增强：`compressed` 模式压缩资源。
- 验证：
  - 在另一台机器直接打开 `index.html` 可浏览全部节点与媒体。
  - 断网状态下导出内容可正常访问。

### 步骤 9：接入原型与动效规范（前端专项）
- 文件：`apps/web/src/features/*`、`docs/ref/*`（建议）
- 操作：将用户提供的原型与动效要求映射为实现任务清单。
- 逻辑：
  - 建立“原型元素 -> 组件/交互”映射表。
  - 对不一致项记录差异并回写规格调整说明。
  - 保证视觉层级、反馈时机、交互动线与原型一致。
- 验证：
  - 与原型逐页比对通过。
  - 关键动效触发与节奏符合要求。

### 步骤 10：质量保障与回归验证
- 文件：`apps/web/tests/*`、`apps/local-server/tests/*`（建议）
- 操作：补齐自动化与手工验收清单。
- 逻辑：
  - API 层：输入校验、错误码、导出完整性测试。
  - 前端层：核心流程 E2E（创建项目 -> 编辑 -> 预览 -> 导出）。
  - 性能层：大节点量与多媒体场景压测。
- 验证：
  - 所有 AC（见 `01_requirement.md`）逐项通过。
  - 导出回归测试稳定通过。

## 实施原则
- 先保证“可用闭环”（新建-编辑-预览-导出），再叠加增强能力。
- 前端体验问题优先级高于非关键增强项。
- 对规格变更采用“先记录、后实现”的方式，避免实现偏离需求。
